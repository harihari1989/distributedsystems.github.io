Build a Postgres-like Database in Python (TUI)

Problem framing

A transactional database must solve correctness, durability, and query performance at the same time. If you write directly to table files without a log, crashes can corrupt state. If you ignore concurrency, readers see inconsistent data. If you skip indexing, every query becomes a full table scan.

This page ships a runnable Python implementation in scripts/mini_postgres_tui.py with three engines: a PostgreSQL-inspired row store, a DynamoDB-inspired key-value model, and a vector database model with ANN-style search. The same TUI can explain all modes: parser, planner/access path, durability model, indexing, and recovery behavior. For broader architecture choices, see storage and data patterns and database patterns.

Core idea / pattern

Layer the engine around a write contract

The central contract is write-ahead logging: commit records are durably persisted before data files are treated as committed state. On restart, replay WAL records newer than the last applied transaction ID. MVCC then defines what each snapshot can see.

Step-by-step flow

1) Parse and validate statements

  1. The parser normalizes a SQL subset: CREATE TABLE, CREATE INDEX, INSERT, SELECT, UPDATE, DELETE, and transaction verbs.
  2. Schema and type checks happen before execution so bad writes fail early.
  3. WHERE clauses are compiled into column/operator/value predicates.

2) Plan read path

  1. If WHERE is a single equality predicate on an indexed column, planner chooses Index Scan.
  2. Otherwise planner falls back to Seq Scan over visible row versions.
  3. EXPLAIN SELECT ... shows the chosen path and estimated rows.

3) Execute writes with MVCC semantics

  1. Each row version stores xmin (creator xid) and xmax (deleter/updater xid).
  2. UPDATE is implemented as delete+insert: old version gets xmax, new version gets new xmin.
  3. VACUUM removes obsolete versions once they are no longer visible.

4) Commit with WAL-first durability

  1. On commit, engine appends BEGIN, one or more OP records, then COMMIT to WAL.
  2. WAL is fsynced before data files are considered committed.
  3. After WAL durability, heap and index files are updated and applied_xid advances.

5) Recover from crash

  1. Startup reads WAL and finds committed transactions with xid greater than last applied_xid.
  2. Missing committed operations are replayed to heap and indexes.
  3. State metadata is updated so replay is not repeated on next boot.

6) Operate through the CLI TUI

The TUI displays command output on the left and runtime internals on the right: counters, storage state, and a concept panel. Use Up/Down for history, Enter to execute, and Esc to exit. 

python3 scripts/mini_postgres_tui.py --data-dir /tmp/mini_pg
python3 scripts/mini_postgres_tui.py --engine postgres --data-dir /tmp/mini_pg --execute "CREATE TABLE users (id INT, name TEXT, balance INT)"
python3 scripts/mini_postgres_tui.py --engine postgres --data-dir /tmp/mini_pg --execute "EXPLAIN SELECT * FROM users WHERE id = 1"

7) Switch the same TUI to DynamoDB mode

Use --engine dynamo to run a DynamoDB-inspired model with partition/sort keys, QUERY vs SCAN behavior, and eventually consistent GSI refresh controlled by TICK. 

python3 scripts/mini_postgres_tui.py --engine dynamo --data-dir /tmp/mini_ddb
python3 scripts/mini_postgres_tui.py --engine dynamo --data-dir /tmp/mini_ddb --execute "CREATE TABLE orders PK user_id SK order_id"
python3 scripts/mini_postgres_tui.py --engine dynamo --data-dir /tmp/mini_ddb --execute "CREATE GSI gsi_email ON orders (email)"
python3 scripts/mini_postgres_tui.py --engine dynamo --data-dir /tmp/mini_ddb --execute "QUERY GSI gsi_email email='a@example.com'"

8) Switch the same TUI to VectorDB mode

Use --engine vector to run a vector database with collections, top-k similarity search, ANN-style partition probing, metadata filters, and explicit index refresh controls. 

python3 scripts/mini_postgres_tui.py --engine vector --data-dir /tmp/mini_vdb
python3 scripts/mini_postgres_tui.py --engine vector --data-dir /tmp/mini_vdb --execute "CREATE COLLECTION docs DIM 4 METRIC COSINE PARTITIONS 8"
python3 scripts/mini_postgres_tui.py --engine vector --data-dir /tmp/mini_vdb --execute "UPSERT docs id=doc1 vector=[0.1,0.2,0.3,0.4] topic=ml"
python3 scripts/mini_postgres_tui.py --engine vector --data-dir /tmp/mini_vdb --execute "SEARCH docs vector=[0.1,0.2,0.3,0.4] TOPK 5 PROBE 3 FILTER topic=ml"

Failure modes

Failure mode What breaks Toy-engine handling PostgreSQL counterpart
Crash after COMMIT WAL, before heap apply Committed rows missing in data file Startup WAL replay by xid Crash recovery redo from WAL
Version bloat from frequent updates Reads become slower, files grow Manual VACUUM command Autovacuum + visibility map
Long-running write burst with indexes Higher commit latency Index rebuild on touched tables Per-index maintenance, HOT optimizations
Type mismatch in writes/filters Incorrect comparisons or silent coercion bugs Strict coercion by schema type Type system + casts + planner checks
No multi-process lock manager Cannot model deadlocks Single-process execution assumption Lock table + deadlock detector

Trade-offs

Design choice Benefit Cost When to evolve
JSONL heap versions Readable, inspectable storage No page cache realism Switch to fixed-size pages and buffer pool
Logical WAL records Easy replay semantics No physical page delta optimization Add page-level WAL for realism
Rule-based planner Deterministic and understandable Poor adaptability to skewed data Add statistics and cost model
Manual vacuum Simple control flow Operational burden and latency spikes Add background maintenance worker
Single-column indexes Fast implementation, clear teaching value No range optimization or composite keys Add B-tree pages and multi-column keys
